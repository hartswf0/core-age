Executive Summary
The "Line 22 Collapse" was a critical data pathology where all studs in a LEGO model appeared to belong to a single MPD line (often Line 22), regardless of their actual origin.

This was not a bug in the stud sampler, but a fundamental entropic destruction of information caused by the monolithic loading architecture (Architecture A).

The solution (Architecture B) involved a "hard structural move" to a Mesh-Per-Line compiler, effectively trading GPU draw calls for informational preservation.

Mechanism of Action
In Architecture A, the entire MPD text was fed to LDrawLoader.parse() as a single string.

The Entropy Event:
1. Parsing: Loader reads text line-by-line.
2. Sorting: Groups geometry by material (color).
3. Merging: Flattens buckets into Float32Array buffers.

Result: "Index != Time"
The resulting geometry is a soup of triangles. It is mathematically impossible to reconstruct the "Line Number" from the position buffer alone.

Explicit Context
We moved from Implicit context to Explicit context.

Mechanism: LineAwareCompiler
Instead of one parse call, we run a loop:
1. Create a discrete Group for THIS line.
2. Explicitly tag userData.lineNum.
3. Load ONLY this part's geometry.

The "Vaccine" (Monkey-Patch):
We also patch THREE.Mesh to ensure even deeply nested meshes catch the context from a global state.
‚úÖ The Structural Load
for (line of mpdLines) {
  const group = new THREE.Group();
  group.userData.lineNum = line.index;
  scene.add(group);
}

The Cost of Truth
The shift to Architecture B was necessary because the application's primary value proposition is Assembly Analysis, not raw rendering performance.

We accepted a ~50x increase in draw calls to gain 100% data integrity.

Verdict: The "Line 22 Collapse" is permanently cured.
Recommendation: Use LineAwareCompiler for all assembly/editing workflows.
